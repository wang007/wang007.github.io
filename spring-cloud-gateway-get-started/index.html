    spring-cloud-gateway 上手

    发表于 2021-02-02 更新于 2021-02-07 分类于 spring webflux ， 异步

    零

    网关在平时微服务中也是比较常用到的一个组件，而作为 spring 系的网关 spring-cloud-gateway 是 java 微服务开发中最常用的网关选项。

    1. spring-cloud-gateway 作为切入点，讲讲 spring-cloud-gateway 简单上手，以及如何开发一个生产可用的网关。
    2. 因为 spring-cloud-gateway 基于 spring webflux 开发，而 spring-webflux 基于 netty 和 reactor 开发。
      所以将出一个系列的文章来逐步分析整个过程的实现原理。

    这里先挖个大坑，这个系列包括的内容

    1. spring-cloud-gateway 上手，实战，实现原理。
    2. spring-webflux 上手，实战，实现原理。
    3. reactor-netty 实现原理（spring webflux 与 netty 的承接层）。
    4. netty 核心原理组件实现原理。

    接下来先说说网关简单上手。

    正文

    先看看 spring cloud gateway 的架构图，经过一系列 filter，然后最终请求 后端服务。
    实际上这个图忽略了一个重要的信息：Route。

    spring cloud gateway 执行流程：

    1. request 请求进来匹配 Route。
    2. 执行 一系列 filter（包括 GlobalFilter，GatewayFilter）。
    3. 请求后端服务。
    4. 将后端服务的响应结果返回给 client。

    所以我们使用 spring cloud gateway 完成网关的功能就是定义一系列 Route，包括配置文件定义，直接写在代码中定义。
    接下来先看看 Route 字段：

    1. predicate: 判断是否匹配当前 Route。入参：ServerWebExchange，就是 request，response 综合体。出参：Publisher<Boolean>，是否匹配成功。
    2. gatewayFilters: 一系列 Filter，用于加工处理 request，response。双向的链式结构，其中入（请求进来）方向最后一个 Filter 是请求后端服务的处理逻辑。

    配置 Route Filter

    1. 添加 request header 到后端服务

    往 request header 添加 X-Request-red: blue, 最终到后端服务里的 request header 包含 X-Request-red: blue。

    这里没有配置 predicates，那么默认就是 always 匹配的，所以正常业务情况， predicates 都要匹配的。

    2. 添加 request 路径参数到后端服务

    往 request 路径中添加参数 red=blue，最终到后端服务里的 request 路径参数包含 red=blue

    常用 API

    1. Route

    关于 Route 前面已经分析过了。

    2. RouteLocator

    用于保存 Route，spring-cloud-gateway 内部就是通过 RouteLocator 获取 Route。

    接口非常简单，就一个 getRoutes 方法获取 Route 集合。
    而 RouteLocator 的实现通常是由我们自己实现，而且经常是提供多个 RouteLocator 的实现给框架内部。

    框架内部使用 CompositeRouteLocator 把多个 RouteLocator 包装成一个 RouteLocator

    最终提供给框架的是 CachingRouteLocator，在 GatewayAutoConfiguration 类中，

    最终使用方创建的所有的 RouteLocator bean 被 CompositeRouteLocator 组合起来，交给 CachingRouteLocator 包装起来，
    而 CachingRouteLocator 就是提供缓存功能的 RouteLocator。

    CachingRouteLocator 带有缓存的功能，所以当动态改变 RouteLocator 的路由信息时，必须广播 RefreshRoutesEvent，实现 Route 缓存刷新。

    广播刷新 Route，然后 CachingRouteLocator#onApplicationEvent 方法收到 Event，然后进行刷新。

    3. RouteLocatorBuilder

    前面说了 RouteLocator，那么 RouteLocator 怎么创建出来呢？
    当然可以直接实现 RouteLocator 接口，提供 Route 信息，但是更便捷的方式是使用 RouteLocatorBuilder。

    就一个方法，通过 routes 方法创建 RouteLocator 构造器。

    1. Builder 使用 route 方法声明式的构造 Route.AsyncBuilder，最终加到 List 容器中。
    2. build 方法：调用 Route.AsyncBuilder#build 方法 转成 Route，最终包装成 Flux<Route>

    4. GlobalFilter

    GlobalFilter 跟 GatewayFilter 一样，最终会加到 GatewayFilter 执行链上。只不过 GlobalFilter 会加到所有 Route 的执行链上。
    所以一般我们定义网关业务逻辑的时候，一般定义到 GlobalFilter
